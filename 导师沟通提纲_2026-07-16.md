# 导师沟通提纲：Graph-level Latency Estimator for Rewrite Plan Selection

## 1. 我准备怎么介绍这个课题

我现在想做的课题不是泛泛的“用 GNN 加速大模型推理”，而是更具体地放在大模型推理系统里的图优化 / 编译优化层。

问题可以这样表述：

> 给定一个 LLM 推理计算图中的局部 block，以及若干语义等价的 candidate rewrite plans，能否在真实编译、lowering、profile 之前，用一个轻量的图级 estimator / selector 预测哪个 plan 更可能快，从而减少搜索和测量成本？

也就是说，这个课题的核心不是直接改 kernel，也不是在线 serving 调度，而是：

> rewrite plan selection / learned cost model for equivalent graph space。

我想强调三点：

1. **问题主张**：rewrite plan selection 不是单纯局部规则问题，因为 rewrite 的收益会受上下游结构、fusion 机会、layout / memory movement、backend lowering 等影响。
2. **方法主张**：先用图结构或结构化特征建模 candidate graph，再做 ranking，而不是只做 latency regression。
3. **评估主张**：主指标应该是 selected-plan latency、regret、win rate、fixed-budget selection quality，而不是 RMSE。

## 2. 当前已经做了什么

我已经先做了一个很小的验证闭环，目的不是证明模型有效，而是验证这个问题是否真实存在。

当前设定：

- block：Transformer MLP / SwiGLU-like block
- IR：Torch FX
- backend：PyTorch eager + `torch.compile(inductor)`，主口径用 Inductor
- dtype：fp16
- batch size：1
- GPU：A100
- rewrite family：gate/up projection 相关 fusion variants

当前候选空间有 6 个 candidate plans：

1. baseline separate gate/up + `F.silu`
2. separate gate/up + manual silu
3. separate gate/up + inplace multiply
4. fused gate/up + chunk + `F.silu`
5. fused gate/up + split + inplace multiply
6. fused gate/up + chunk + manual silu

已完成的验证：

- 这些 candidates 都能实例化为可执行 PyTorch module；
- 都能 trace 成 Torch FX graph；
- 和 baseline 数值等价；
- 实际 FX graph 结构签名不重复；
- 已经接入 `torch.compile(inductor)` profiling；
- 有 raw profile、candidate summary、CV、winner flip 统计。

## 3. 当前初步结果

目前只测了 3 个 block，所以结果只能当作初步信号。

在 compile backend 下：

| block | baseline p50 | best candidate | best p50 | spread | winner flip |
| --- | ---: | --- | ---: | ---: | --- |
| b0 seq128 h1024 i4096 | 0.096768 ms | p5 fused chunk manual silu | 0.086016 ms | 12.50% | false |
| b1 seq128 h768 i3072 | 0.097280 ms | p3 fused chunk silu | 0.089600 ms | 12.57% | false |
| b2 seq512 h768 i3072 | 0.108544 ms | p3 fused chunk silu | 0.104448 ms | 3.92% | true |

我的理解：

- 这个问题不是完全空的：候选之间确实有 latency spread；
- b0 / b1 信号还可以，winner 不翻转；
- b2 比较危险，spread 小且 winner flip，说明噪声已经接近候选差异；
- 当前 candidate space 太窄，oracle ceiling 不够高；
- 现在还不能说图模型有必要，更不能说 GNN 有效。

## 7. 我建议的下一步路线

如果导师没有强烈反对，我建议继续按下面流程推进：

1. 先做 simple heuristic baseline  
   看 fusion-prior 等简单规则是否已经接近 oracle。

2. 同时扩 candidate space  
   目标是提高 oracle ceiling，让 candidate spread 稳定大于 profiling noise。

3. 扩 block / shape  
   当前只有 3 个 block，必须扩到几十个配置，至少覆盖 seq=128 / 512 / 2048。

4. 做弱学习器  
   先做 handcrafted features + XGBoost / MLP，不要直接上 GNN。

5. 如果简单模型做不完，再上结构化模型  
   可以考虑 linearized graph encoder、DAG-aware message passing、轻量 GNN。

6. 最后再考虑 chitu 轻量验证  
   如果方法站稳，再把它接到真实系统环境里做系统层证明。
