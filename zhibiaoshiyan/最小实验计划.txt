  ====================
  二、实验目标
  ====================

  这次实验只回答 4 个问题：

  1. candidate rewrite plans 之间是否存在真实、稳定的 latency 差异？
  2. baseline/default plan 距离 candidate set 内最优 plan 有多远？
  3. 一个极简单的 selector（先不用 GNN）能不能获得稳定收益？
  4. selector 本身的 decision overhead 是否足够低，不会吃掉收益？

  ====================
  三、固定实验设定
  ====================

  请尽量按下面设定执行；如果必须调整，请在最终报告里明确说明调整原因。

  1. block 类型
  - 优先选择 Transformer MLP block
  - 如果当前代码/模型抽取不方便，也可以选择最接近的高频 block，但必须说明替代原因

  2. backend
  - 单一 backend
  - 单卡 GPU
  - 固定 dtype（fp16 或 bf16，按环境最稳定者）
  - 固定 batch_size = 1

  3. shape 设定
  优先使用 3 档 sequence length：
  - 128
  - 512
  - 2048

  如果某些 shape 不可运行，可以替换成当前环境稳定可跑的 3 档，并在报告中写清楚。

  4. graph / block 实例数
  目标：
  - 总 graph 数 = 60

  推荐：
  - 每个 shape 20 个 graph
  - 总共 60 个 graph

  如果当前只能拿到更少实例，最低不能少于 30 个 graph，并在报告中说明原因。

  5. candidate plan 数
  目标：
  - 每个 graph 生成 4~8 个 candidate plans
  - 优先固定为每图 6 个 candidate

  每个 graph 必须至少包含：
  - 1 个 baseline/default plan
  - 至少 3 个非 baseline 候选

  6. rewrite family
  第一版只做 1 类 rewrite family，优先：
  - fusion-related
  或
  - elementwise reorder
  或
  - transpose/layout movement

  请选当前环境里最容易稳定实现的一类，不要一次混多类。

  ====================
  四、selector 设定
  ====================

  这次不要做复杂学习模型，先做 3 种 selector：

  A. Baseline selector
  - 永远选默认 plan / 当前系统已有 plan

  B. Heuristic selector
  - 使用简单规则打分并选择
  - 规则可以来自：
    - transpose 越少越好
    - estimated bytes 越少越好
    - fusion 越多越好
    - elementwise chain 越短越好
    - 节点数/边数/简单图统计
  - 目标是做一个轻量、可解释、无需训练的 selector

  C. Oracle selector
  - 使用真实测量 latency 最小的 candidate 作为 oracle
  - 注意：oracle 只用于评估上界，不作为部署方法

  ====================
  五、测量协议
  ====================

  1. candidate latency 测量
  对每个 (graph, candidate)：

  - warmup = 10 次
  - 正式测量 = 30 次
  - 记录每次 latency_ms

  如果噪声很大，可把正式测量提升到 50 次，但要在报告里说明。

  2. 每个 candidate 的汇总统计
  至少保存：
  - p50 latency
  - mean latency
  - std latency
  - cv = std / mean

  3. decision time per graph 测量
  定义：
  - 从输入 graph/candidate 特征到输出最终选中的 candidate ID 的时间

  必须包含：
  - graph 特征提取
  - candidate 特征组织
  - heuristic 打分
  - 排序和选 top-1

  不得包含：
  - 候选真实 latency 测量
  - 离线训练
  - compile
  - 离线数据生成

  测量方式：
  - 对每个 graph 的 selector 决策过程重复 100 次
  - 记录每次 decision_time_ms
  - 最终保存：
    - p50 decision time
    - p95 decision time

  4. win 定义
  请固定以下定义：

  - 若 selected plan 相对 baseline plan 提升超过 2%，则记为一次 win
  - 即：
    if selected_latency <= 0.98 * baseline_latency:
        win = 1
    else:
        win = 0

  ====================
  六、核心指标定义
  ====================

  请严格按下面定义计算指标：

  1. selected-plan latency speedup
  - speedup_vs_baseline = baseline_latency_ms / selected_latency_ms

  2. oracle gap / regret
  - regret_vs_oracle = selected_latency_ms / oracle_latency_ms - 1

  3. decision time per graph
  - 使用 decision_time_p50_ms 作为主口径
  - decision_time_p95_ms 作为补充口径

  4. win rate
  - win_rate = mean(win_over_2pct)

  十、失败/止损判断
  ====================

  如果出现以下情况，请不要盲目继续堆复杂方法，而是先在报告中明确诊断：

  1. speedup 接近 1.00x，win rate 接近 50%

  3. heuristic 与 baseline 基本无差别，regret 很大
  - 说明当前特征不够，或者问题确实需要更强结构信息

  4. decision time 明显过高
  - 说明 selector 成本可能抵消收益
