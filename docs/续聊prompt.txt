====================
一、对话总结
====================

我当前的主课题已经基本确定为：
Graph-level latency estimator for rewrite plan selection

目标是：
在 2026 年 10 月前，做出一篇可投稿 ICLR 的工作。

这不是一个泛泛的“用 GNN 加速推理”的题，而是更具体地落在：
大模型推理系统中的图优化 / 编译优化层。

核心问题不是 runtime 调度，也不是在线 serving 调度，而是：
给定原始计算图和若干语义等价的 candidate rewrite plans，在真实执行前预测不同 plan 的图级执行效果，并在有限搜索预算下选到更优 plan。

我们已经明确过的关键判断如下：

1. 课题定位
- 这是一个 rewrite plan selection 问题，不是简单 latency regression 问题。
- 真正要优化的是 selection quality，而不是只追求预测毫秒数更准。
- 更合适的任务形式是 ranking-oriented，尤其是 pairwise ranking，可辅以 delta-latency regression。

2. 为什么这个题成立
- rewrite 的真实性能收益往往不是局部决定的，而会受到图级结构、上下游邻域、layout 传播、fusion 机会、memory movement、lowering 结果等共同影响。
- 因此“单算子代价求和”或“局部启发式”可能不够，需要显式建模图结构或至少建模作用域子图。
- 但不能先验假设 GNN 一定更好，必须靠实验验证；图模型只是一个有根据的候选方法，不是结论。

3. 推荐的方法主线
- 输入对象：候选 rewrite 后得到的 candidate graph，或 rewrite 作用域扩张后的局部图。
- 目标：对候选 plans 打分或排序。
- 主任务：pairwise ranking。
- 辅助任务：delta-latency regression。
- 评价重点：fixed-budget 下的 plan selection quality，例如 top-1 accuracy、regret、固定预算下选中 plan 的 latency。

4. 系统位置
- 该方法更适合插在：候选 rewrite plans 枚举之后、真实编译 / lowering / autotuning / profile 之前。
- 它本质上是 graph optimizer 里的 rewrite-plan ranking / cost-model 层。
- 因此它仍可算作“大模型推理服务相关课题”，但更准确地说，是服务系统栈里的离线图优化 / 编译优化层方法，而不是在线请求调度层。

5. 范围收缩
为了保证 ICLR 截稿前能闭环，当前建议的最小可行范围是：
- 单卡 GPU
- 单一 backend
- 离线实验
- 只选 1 到 2 类高频 block
- 只做少量 rewrite family

优先 block：
- Transformer MLP block
- Attention 相关 block

优先 rewrite family：
- fusion-related rewrites
- transpose / layout movement
- reassociation / elementwise reorder

暂时不做：
- 多 backend 联合建模
- 全模型任意 rewrite
- 大规模 e-graph 全空间搜索
- 在线 runtime 集成
- 多机多卡实验

6. 数据与实验框架
计划是：
- 从开源 LLM block 中抽取高频子图
- 转成统一 IR（如 ONNX / Torch FX / Relay 中的一种）
- 枚举有限个合法 rewrite plans
- 对每个候选做真实编译与 profile
- 形成 (原图, 候选 plan 图, 条件, latency / ranking label) 数据

baseline 至少包括：
- 规则启发式
- op-wise latency sum / FLOPs+bytes proxy
- graph summary + MLP
- handcrafted features + XGBoost
- linearized graph + Transformer/MLP
- oracle best plan

关键指标包括：
- pairwise accuracy
- Spearman / Kendall tau
- top-k hit rate
- top-1 plan selection accuracy
- regret relative to oracle
- fixed-budget selected-plan latency
- search cost reduction

真正关键的问题是：
在相同搜索预算下，能否稳定选到更快的 rewrite plan。

7. 风险判断
已经明确过的主要风险有：
- 图模型可能不显著优于简单模型
- latency 标签噪声较大
- 前端 rewrite 差异可能被后端 lowering 抹平
- 审稿人可能觉得问题过于工程化

对应的应对策略包括：
- 优先选择图级交互更强的 rewrite family
- 做强 baseline，避免虚假优势
- 固定 profile 环境，多次测量取中位数
- 先验证 candidate diversity 是否真实存在
- 强调 equivalent graph space 上的 learning decision 问题，而不是某个系统 patch
